The CSS included in Headstart is powered by Sass, and forces next to no default styles into your workflow. With just a few easy to use mixins and functions you can make any kind of layout happen!

Folder structure

Inside of the app folder you will find the sass folder. This framework uses the SCSS syntax by default, as it resembles good ol' CSS and makes it easier to include regular css files (albeit with a different extension). Feel free however to put in your own Sass files, if you're already satisfied with your setup. Headstart supports both .scss and .sass files, as long as they are placed inside of the sass folder.

Great Sass comes with great responsibility. By separating the responsibility of your styles into named chunks, finding out where a certain style might be getting applied becomes a lot easier. This is the setup that you are presented with by default:

  • ./base ~ Files that do not generate CSS on their own.
  • ./mixin ~ CSS generating modules.
  • ./module ~ Chunks of styles that are separated because of their specific purposes.
  • ./vendor ~ All third-party CSS that comes with eg. plugins.
  • ./ ~ The root files combine the different chunks and contain other, loose styles.

Base files

All files in these folder don' actually generate CSS on their own.

_config_global.scss

The global config file is something you usually won't have to change a lot. If you are used to using a library, you can import those here.

This is also where a bunch of high-level variables can be configured to your likings:

  • $base-font-size ~ Defaults to 16px, the de facto browser font size.
  • $total-columns ~ Defaults to 12, which devides easily for 2-, 3- and 4-column layouts.
  • $default-gutter-width ~ Defaults to 5% and adds fluid spacing in between columns.
  • $default-column-breakpoint ~ Defaults to 650px and indicates which space will make columns start floating.
  • $max-container-width ~ The maximum width a container can be.
  • $min-page-padding ~ When the available space is less wide than the $max-container-width, this is the amount of side-padding that will be added.

At the end of this file, all mixins and functions are being imported to make them available for all files that import this config later on.

Don't forget that your files will be getting auto-prefixed. If you have been using Compass for vendor-prefix purposes, consider switching to regular CSS3. Otherwise, turn of this feature in the gulpfile.js's config.

_config_site.scss

This file defines variables that pertain to project specific settings, such as accent colors, shades of black/white and the font-stack. The latter defaulting to a solid, cross-browser sans-serif setup:

$font-stack: Helvetica Neue, Helvetica, Verdana, Roboto, sans-serif;

_functions.scss

Some bare-bones functions that will really come in handy:

  • strip($number) ~ Will string the unit from a passed value.
  • em($target, $context: $base-font-size) ~ Generates em values from a number. Supports negatives as well. The $context variable is used as base value during calculations.
  • rem($target) ~ Generates rem values with px fallbacks from a number. Also supports negatives.
  • tint($color, $percent) ~ Will lighten a color with a percentage of white.
  • darken($color, $percent) ~ Will darken a color with a percentage of black.

Mixins

Mixins will generate pieces of code so they can be reused easily.

_all.scss

A wrapper for easily importing all available mixin modules at once. This file gets imported into _config_global.scss by default.

_grid.scss

The grid module allows you to easily make auto-collapsing columns.

/* column($columns, $end: false, $break: $default-column-breakpoint, $context: $total-columns) */

aside {
/* Make it float left and take up 4 of 12 columns */
@include column(4);
}
article {
/* Make it close the row by floating right and taking up 8 columns */
@include column(8, true);
}

By default, a column will start being a column when the available width is at least $default-column-breakpoint or 650px, which gets defined in the base files.

You can also have multiple column settings by wrapping the styles in media queries (more on those in the next chapter):

/* Suppose you have 4 divs with content */
/* Example code that is actually used on the homepage */

div {
/* Setup a 2 column layout */
/* Starting at 650px (default) up to 1049px */
@include media-max(1049px)
{
/* Odd columns (1,3,5,...) will float left */
&:nth-child(odd) { @include column(6); }
/* Even columns (2,4,6,...) will float right */
&:nth-child(even) { @include column(6,true); }
/* First two blocks push down */
&:nth-child(-n + 2) { margin-bottom: rem(50); }
}

/* Set up a 4 column layout */
/* Starting at 1050px */
@include media-min(1050px)
{
/* All but the last column float left */
&:not(:last-child) { @include column(3); }
/* Last column floats right */
&:last-child { @include column(3,true); }
}
}

If the default breakpoint is good for most of your site, but you want one block to have a later entry point, you can add a custom breakpoint as well:

/* The default value of 650px will be dropped in favour of 800px */
div { @include column(6, false, 800px); }

But won't nesting and repeating media queries at element level generate a lot of extra code? Isn't combining media queries better for performance?
Turns out it doesn't. But the choice is yours! If you feel like combining them at the bottom or in a separate module, go ahead!

_media.scss

Headstart comes with 4 media query mixins that make it a painless process to tie styles to certain dimensions.

/* Un until given value */
@include media-max($value) {
...
}

/* Starting from given value */
@include media-min($value) {
...
}

/* Between given min and max */
@include media-min-max($valueMin, $valueMax) {
...
}

/* On retina screens */
@include highres() {
...
}
  • Passed in values will be stripped of their unit.
  • All media-* functions have an optional $query parameter that defaults to 'screen'.

_prop.scss

The property related module only contains one mixin, rem(). It generates rem values with pixel fallbacks for a certain property. Great for paddings and margins!

/* For example, using: */

div {
@include rem(margin-top, 20);
@include rem(padding, 10, 20, 10, 30);
}

/* Will result in */

div {
margin-top: 20px;
margin-top: 1.25rem;
padding: 10px 20px 10px 30px;
padding: 0.625rem 1.25rem 0.625rem 1.875rem;
}

Not to be confused with the likewise named function, which doesn't provide fallback values.

_triangle.scss

CSS triangles, we all love them, but loathe setting them up. This mixin makes it easy to construct a triangle in a split-second.

/* triangle($width: 10px, $height: 10px, $direction: up, $color: red) */

.tooltip:before {
content: '';
display: block;
@include triangle(15px, 10px, down, #fff);
...
}

Modules

Modules are specialized chunks of styles that are separated to lighten your main files. Especially useful when combining Sass placeholder classes with @extend. Placeholder classes also don't output any CSS on their own, which is nice!

_button.scss

Define all your button styles in here. The placeholder class %button normalizes some properties that are set on the most common button-elements, whereas modifier placeholders—eg. %button--stroke— target specifically styled buttons.
The latter ones extend the main placeholder, and can be extend from again by your own buttons, which make the total very readable and DRY:

/* Main placeholder */
%button {
/* Resets here */
...
}

/* Modifier placeholder */
%button--stroked {
@extend %button;
border: 1px solid #000;
}

/* Somewhere in your main code */
.contact-submit-button {
@extend %button--stroked;
}

Above setup would require only one—readable and understandable—class in your HTML, and through the power of Sass, selectors will be combined to the absolute minimum.

_container.scss

Containers are centered blocks with a maximum width. When the available width is less than their defined maximum, they get padded on both sides. Images inside of these type of blocks are made fluid by default so they scale along beautifully.

/* Main class */
.container {
...
img { ... }
}

/* Some semantically named container */
.container--header {
@extend .container;
background: #000;
}

_font.scss

A place to put your @font-face configuration. Contains a commented-out dummy so you don't forget to include any fallbacks.

_form.scss

Includes some default, normalized styles for forms elements

_normalize.scss

Resetting elements is something of the past. Making elements render more consistently across browsers. This is basically just the Normalize.css library.

_print.scss

The print module is limited to just a few styles that will improve the printing "experience" of your website, eg. making all the text black, removing background-images and shadows, etc. This file is imported into the main stylesheet to prevent an extra GET request.

_reporting.scss

This module makes it easy to add reporting paragraphs and lists. It is also used by the Utils class for usage through script.

Add a info-block with the class .report--info.

Add a warning-block with the class .report--warning.

Add a success-block with the class .report--success.

  • Lists are also supported.
  • Display errors with the .report--error class.

_state.scss

This files contains a bunch of handy (silent) classes that make it easy to keep your stylesheets DRY:

  • %is-hidden ~ Visually hide an element. Handy for image replacements.
  • %is-bold
  • %is-underlined
  • %is-italic
  • %is-uppercase
  • %is-smallcaps
  • %is-cf ~ Clears floats. Can also be used as stand-alone class by default.
  • %is-vertical-aligned ~ Alignes stuff vertically in its parent.

_text.scss

Everything text-related is contained in this module:

  • Text-selection color
  • Headings
  • Lists
  • Paragraphs
  • Links

Vendor files

The vendor folder is where you can put the files that come with the plugins you use. Since Headstart uses SCSS, just change the .css extension to .scss and you're good to go.

Root files

Common styles

This stylesheet contains the main styles that are used on each and every page. For example the styles of your <header>, <menu> or <footer> elements.

In the Lite version of Headstart this stylesheet is called styles.scss.

View specific styles

When working on a large scale project, somes styles will be limited to individual pages. This is when the view-specific files come in handy, as they will lighten up the main stylesheet and thus speed up the individual pages of your website.

  • View-specific sheets are only used in the Main version of Headstart.
  • These files should be prefixed with view-, followed by the name of the target HTML file. For example view-index.scss, view-about.scss or view-portfolio.scss
